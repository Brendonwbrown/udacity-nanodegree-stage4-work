Introduction to Networks

    • Networks

    • A network is a group of entities that can communicate, even though they are no all directly connected.
    • The internet is a special kind of network.
    • The message has to be encoded in a way that all nodes understand
    • Routing instructions - send messages between specific points requires instructions to notes passing the message on.
    • Priority and permissions on the network.

    • Measuring Networks

    • Latency is the time it takes to send a message from source to destination.
    • Bandwith is the amount of information which can be transmitted per unit of time.
    • Bandwidth / time is measured for the internet in bits/second.
    • 1 bit is the answer to 1 yes/no question, or 0/1.

    • Protocols

    • A protocol is the set of rules used by a client to communicate with a server.
    • Web protocol is Hypert Text Transfer Protocol
    • Browser submits a web request to server, server sends the answer.

    • HTML Document Structure

    • Include a simple doctype
    • The head includes a TITLE, some META items and LINK items to style and script.
    • Most everything else goes in the BODY section.
    • URLs are composed of the PROTOCOL, HOSTNAME and PATH.
    • A URL can also contain some QUERY information, which includes a question mark and NAME=PARAMETER. Multiple queries are strung together with & .
    • A FRAGMENT is used to reference a part of the page. Not sent to the server - last in the URL preceded by #
    • A PORT goes right after the HOSTNAME. It's necessary to make the connection to the host - by default, HTML goes over 80.

    • HTTP request and response lines.

    • The HTTP protocol is the protocol used to talk with a web server.
    • A request line contains a METHOD, then PATH, then HTTP VERSION
    • METHOD can use GET or POST, VERSION can be 1.0 or 1.1
    • A request line is followed by HEADERS. NAME:VALUE. "Host:www.example.com, User-Agent:Chrome
    • Requesting info from the server is only part of the equation, the server also responds in its own way
    • The response comes in a STATUS LINE containing an HTTP VERSION, STATUS CODE and REASON PHRASE.
    • Status codes can begin with 2 through 5.
    • 200 means status is OK, 302 when the resource is found, 404 when a resource is not found, and 500 when there's a server error.
    • HEADERS also are returned: date, server software, content type and content length. Sending back server software is not a great idea, and not all returned headers are relevant.

    • Web applications Intro

    • A client can request static or dynamic content.
    • There's static html pages built composed of text, images and documents.
    • There are other pages built on the fly and compiled into browser-readable information with server-side software.

Introduction to Forms

    • What are Forms?

    • Use the FORM tag to introduct a form.
    • The INPUT tag takes some form of user content. Add a NAME attribute.
    • INPUT tags take a TYPE attribute which drastically assists building a form.
    • TEXT, CHECKBOX, SUBMIT, RADIO...
    • Add a LABEL element around the INPUT or use the FOR parameter to assign it.
    • A SELECT tag containing OPTION tags creates a dropdown list.

Some Additional Tools

    • Modulo

    • Modulo (%) gives the remainder. Divide something by something. If there's leftover...

    • Dictionary

    • There are strings "..."
    • There are lists = ['brendon', 'wayne','brown']
    • There are dictionaries = {'first':'brendon','middle':'wayne','last':'brown'
    • Dictionaries will return the value when the key is invoke like so: d[k]
    • Or 'print KEY in DICTIONARY'
    • Update a dictionary with DICTIONARY['KEY']=VALUE
    • A Dictionary can contain a whole dictionary! Return contents with multiple square brackets and keys: DICTIONARY[KEY][KEY]

Validation

    • What is validation?

    • Not everyone who interacts with our website will be good intentioned - allowing input means making our site vulnerable.
    • We also need to ensure well-intention users enter relevant content. People make mistakes. We can help them enter numbers in the number fields and emails in the email fields.
    • HTML has some validation built in. Otherwise we can use Python to validate input content on the server side.
    • A user sends a GET request. The server returns form data. User POSTs the form information. Server validates POST on the way in and either approves or returns form data and an error.
    • Verify input, detect errors, render old form with additional error message.
    • Don't forget to "escape" some of the characters that could mess up your HTML.
    • Redirect user to a new page when they are successful.
    • Not validating input could allow someone to inject their own code to steal authentication information, or at best, code to mess up site visual appearance.

Templates

    • HTML Templating

    • A template library is a library to build complicated strings in HTML.
    • Using templates allows for building complex applications, like blocks added to scaffolding.
    • This also allows us to reuse chunks of HTML on several different pages. This also means that if we need to make a change reflected on all the pages, we only need to make it once.
    • Templating allows for easier debugging and updating.
    • JINJA!
    • Variable substitution with {{ curly braces }}
    • Statements with {% Statement %} {% End Statement %}
    • For loops: {% for statement %} {% end statement %}
    • Templating plugins should include auto escaping stuff.
    • Minimize code in templates.
    • Minimize html in code.

Databases

    • What is a database.

    • A program that stores and retrieves large amounts of data.
    • A database can also refer to a machine or system of machines hosting this data.
    • The browser asks the server for infomation, so the server calls it up from a database.

    • Tables

    • A database is composed of tables.
    • Tables are composed of columns and rows.
    • Columns have types: integer, date, string (varchar)...

    • Why Databases?

    • Querying data by hand is tedious, slow and processor intensive.
    • SQL databases are relational: Postgresql, Mysql, Sqlite, Oracle...
    • Google AppEngine Datastore
    • Amazon Dynamo
    • NoSQL, MongoDB, Couch...
    • There are plenty of options - choose the one that fits the situation best.

    • SQL

    • Structured Query Language
    • Invented in the 1970s
    • SELECT * FROM links WHERE id = 5
    • Indexes significantly speed up read times. Hashtables can be used for indexes, each entry is about the same time to lookup,but they lose sorting info. Trees are sorted, but lookups are slower and they vary in time.

    • ACIC

    • Atomicity - All parts of a transaction succeed or fail together. Don't allow adding a row without ALL columns filled.
    • Consistency - consistent databasing - update the score, update the user's karma.
    • Isolation - no transaction can interfere with another's. An upvote and downvote cannot done via the same transaction.
    • Durability - Once the transaction is committed it will not be lost. Backups...

    • Google App Engine Database

    • Tables are called entities.
    • Columns are not fixed.
